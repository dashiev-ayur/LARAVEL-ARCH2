# Подключение Swagger

Ниже — пошаговое подключение интерактивной документации OpenAPI (Swagger UI) в Laravel через пакет [L5-Swagger](https://github.com/DarkaOnLine/L5-Swagger). Для Laravel 13 используйте актуальную ветку пакета (11.x): она рассчитана на `laravel/framework ^13`, PHP 8.2+ и `zircote/swagger-php` 6.x.


## Как это работает в связке Laravel и Swagger

**OpenAPI** — это формат описания HTTP API (пути, методы, параметры, тела запросов, ответы, схемы данных, безопасность). Файл спецификации обычно в JSON или YAML. Исторически тот же класс документов называли **Swagger**; в экосистеме часто говорят «Swagger» имея в виду и спецификацию OpenAPI, и инструменты вокруг неё.

**Swagger UI** — отдельное веб-приложение (статический фронтенд). Оно не «знает» Laravel: ему достаточно URL или встроенного текста спецификации. UI читает OpenAPI-документ и рисует страницу с описанием методов; кнопка «выполнить запрос» отправляет реальные HTTP-запросы на тот хост, который указан в спецификации или в настройках серверов (`servers`).

**swagger-php** (`zircote/swagger-php`) — PHP-библиотека, которая **собирает** OpenAPI-документ из вашего кода. Она обходит указанные каталоги, читает классы и методы (через рефлексию) и интерпретирует **атрибуты** или аннотации в стиле OpenAPI. Маршруты Laravel при этом **не сканируются автоматически**: соответствие «роут в `routes/web.php` / `api.php`» и «описание в атрибутах» вы поддерживаете сами, иначе документация разойдётся с фактическим API.

**L5-Swagger** — обёртка для Laravel: регистрирует сервис-провайдер, публикует конфиг и шаблоны, добавляет Artisan-команду генерации и маршруты, по которым отдаётся Swagger UI и сгенерированная спецификация. Команда `l5-swagger:generate` вызывает swagger-php с настройками из `config/l5-swagger.php`, записывает результат (например, в `storage/api-docs`) и подключает это к UI.

**Поток данных:** вы описываете API в PHP (атрибуты на контроллерах или в отдельных файлах) → при генерации swagger-php строит один цельный OpenAPI-файл → Laravel отдаёт его и страницу Swagger UI → браузер показывает документацию и может дергать ваше приложение по HTTP.

Генерация спецификации — **отдельный шаг** от обработки обычных запросов приложения: до вызова `l5-swagger:generate` (или автогенерации при запросе) изменения в коде в UI не появятся. Спецификацию можно хранить только как артефакт сборки/деплоя или коммитить в репозиторий — это решение команды.



## Практические шаги

1. Установите зависимость из корня проекта:

   ```bash
   composer require darkaonline/l5-swagger
   ```

2. Опубликуйте конфигурацию и шаблоны пакета (начиная с L5-Swagger 6 это стандартный способ):

   ```bash
   php artisan vendor:publish --provider="L5Swagger\L5SwaggerServiceProvider"
   ```

   После публикации появится `config/l5-swagger.php` и при необходимости представления в `resources/views/vendor/l5-swagger`.

3. Укажите в `config/l5-swagger.php`, какие каталоги и файлы сканирует генератор (ключи вроде `annotations` / `scanOptions`), чтобы OpenAPI-описания подхватывались из ваших контроллеров или отдельных PHP-файлов с атрибутами/аннотациями swagger-php. По умолчанию часто сканируются `app` и связанные пути — сверьтесь с опубликованным конфигом после шага 2.

4. При необходимости задайте базовый URL API для спецификации через переменные окружения (имена смотрите в опубликованном `config/l5-swagger.php` и `.env.example`, если вы их добавите). Часто используют константу вида `L5_SWAGGER_CONST_HOST` или аналог из конфига, чтобы в UI корректно подставлялся хост (локально и на стендах).

5. Сгенерируйте JSON/YAML спецификацию и статику UI:

   ```bash
   php artisan l5-swagger:generate
   ```

   Повторяйте команду после изменения атрибутов или аннотаций в коде. Для разработки можно включить автогенерацию при каждом запросе (в `.env`): `L5_SWAGGER_GENERATE_ALWAYS=true` — **не включайте это в продакшене** без понимания нагрузки и кэширования.

6. Запустите приложение (`php artisan serve`, Sail и т.д.) и откройте интерфейс Swagger UI по маршруту из конфигурации (типичный путь по умолчанию — `/api/documentation`; точное значение — в `config/l5-swagger.php`, ключ `routes` / `docs`).

7. Ограничьте доступ к документации в production: оставьте UI только за авторизацией, базовой авторизацией веб-сервера или отключите маршруты документации на публичных окружениях, если спецификация не должна быть открыта.

8. Для `swagger-php` 6.x в PHP предпочтительны **атрибуты** (`OpenApi\Attributes\...`) вместо устаревших DocBlock-аннотаций; синтаксис и примеры — в [документации zircote/swagger-php](https://zircote.github.io/swagger-php/) и примерах репозитория L5-Swagger.


## Команды (шпаргалка)

| Действие | Команда |
|----------|---------|
| Опубликовать конфиг и представления L5-Swagger | `php artisan vendor:publish --provider="L5Swagger\L5SwaggerServiceProvider"` |
| Собрать спецификацию из атрибутов/аннотаций в сканируемых путях | `php artisan l5-swagger:generate` |
| Проверка генерации в тестах | `php artisan test --compact tests/Feature/L5SwaggerGenerateCommandTest.php` |

После генерации JSON по умолчанию лежит в `storage/api-docs/` (имя файла — в `config/l5-swagger.php`, ключи `docs_json` / `docs_yaml`).

**Где открыть в браузере** (значения из `config/l5-swagger.php` для документации `default`; хост и порт подставьте свои):

| Что | Путь | Пример при `php artisan serve` (порт 8000) |
|-----|------|---------------------------------------------|
| **Swagger UI** | `api/documentation` | [http://127.0.0.1:8000/api/documentation](http://127.0.0.1:8000/api/documentation) |
| **OpenAPI JSON** (сырая спецификация) | `docs` | [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs) |

Если приложение доступно по другому базовому URL (Sail, Valet, другой порт), замените только схему, хост и порт; пути `/api/documentation` и `/docs` остаются теми же, пока не меняли конфиг (`documentations.default.routes.api` и `defaults.routes.docs`).

Опционально в `.env`: `L5_SWAGGER_GENERATE_ALWAYS=true` — пересборка при каждом запросе к документации (только для локальной разработки). Базовый хост для подстановки в спецификацию: `L5_SWAGGER_CONST_HOST` (см. `config/l5-swagger.php`, секция `constants`).


## Заглушка в корневой спецификации

В проекте файл `app/Http/OpenApi/OpenApiSpec.php` задаёт минимальный валидный OpenAPI-документ для swagger-php:

- атрибуты **`Info`** и **`Server`** (без `Info` генератор завершится ошибкой);
- один описанный путь **`GET /api/health`** на методе-заглушке `openApiHealthPlaceholder()`.

Отдельного маршрута Laravel для `/api/health` может не быть: это **только описание для спецификации**, чтобы выполнить требование swagger-php о наличии хотя бы одного **PathItem**. Когда появится реальный эндпоинт проверки работоспособности или другой первый метод API, перенесите атрибуты на соответствующий контроллер и удалите или замените заглушку, чтобы документация совпадала с фактическими маршрутами.


Если `composer require` сообщает о несовместимости версий, обновите ограничения фреймворка в `composer.json` или установите версию L5-Swagger, явно совместимую с вашей веткой Laravel (см. [Installation & Configuration](https://github.com/DarkaOnLine/L5-Swagger/wiki/Installation-&-Configuration) на GitHub).
