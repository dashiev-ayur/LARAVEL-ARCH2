# План выката production в Docker

Пошаговый чеклист: от образа до работающего окружения. Предполагается типичный стек Laravel (PHP-FPM + веб-сервер, БД, при необходимости Redis и очереди).

---

## 1. Подготовка репозитория и артефактов

1. Зафиксировать версии: PHP (в проекте — `^8.3`), расширения PHP, версии Node для сборки фронтенда, тег образа БД.
2. Убедиться, что `composer install` для prod выполняется с `--no-dev --optimize-autoloader`.
3. Собрать фронтенд для production (`npm ci` / `pnpm install --frozen-lockfile` и `npm run build` или аналог из `package.json`).
4. Исключить из образа лишнее: `.git`, тесты, dev-зависимости, локальные `.env` — через `.dockerignore`.

---

## 2. Dockerfile приложения

1. Выбрать базовый образ (официальный `php` с нужными расширениями или готовый шаблон под Laravel).
2. Многостадийная сборка (рекомендуется):
   - стадия **build**: Node — только для сборки Vite/React;
   - стадия **runtime**: PHP-FPM (или FrankenPHP), копирование уже собранных `public/build` и `vendor`.
3. Установить системные зависимости и PHP-расширения, которые требует Laravel и драйвер БД (`pdo_pgsql` и т.д.).
4. Скопировать код, выполнить `composer install --no-dev` на стадии, где доступен Composer (или копировать готовый `vendor` из CI).
5. Права на `storage` и `bootstrap/cache`: каталоги должны быть доступны пользователю процесса PHP.
6. Точка входа: либо только PHP-FPM (тогда nginx в отдельном сервисе), либо единый образ с nginx + php-fpm через supervisor — на выбор, но для prod чаще разделяют сервисы.

---

## 3. Compose (или оркестратор) для production

1. Описать сервис **app** (PHP-FPM) с образом из шага 2.
2. Описать **web** (nginx или Caddy): корень документа — `public`, проксирование PHP на сокет/порт FPM.
3. Подключить **БД** (например, PostgreSQL): отдельный сервис, именованный volume для данных, переменные из секретов, не хранить пароли в образе.
4. При использовании очередей и кэша — сервис **Redis** (или managed Redis вне Docker).
5. Отдельный сервис **queue worker** (`php artisan queue:work`) с политикой перезапуска и лимитами памяти.
6. **Scheduler**: один контейнер с `php artisan schedule:work` (Laravel 11+) или cron внутри выделенного сервиса — один экземпляр на приложение, чтобы задачи не дублировались.
7. Сеть: внутренняя сеть compose; наружу только 80/443 (и при необходимости SSH для админов — по политике безопасности).

---

## 4. Переменные окружения и секреты

1. Создать production `.env` (или inject через Docker secrets / переменные оркестратора): `APP_ENV=production`, `APP_DEBUG=false`, уникальный `APP_KEY`.
2. `APP_URL` с реальным HTTPS-доменом.
3. Настройки БД: хост = имя сервиса в compose, не `127.0.0.1`, если БД в другом контейнере.
4. Настроить драйвер сессий, кэша, очереди и почты под prod (часто Redis + SMTP или API провайдера).
5. Не коммитить секреты; для CI/CD — хранилище секретов.

---

## 5. Персистентность и логи

1. Volume для загрузок пользователей (`storage/app`), если файлы не вынесены в S3/объектное хранилище.
2. Логи: stdout/stderr контейнеров или драйвер логирования хоста; ротация на уровне инфраструктуры.

---

## 6. Первый запуск и миграции

1. Поднять БД и дождаться готовности (healthcheck в compose).
2. Запустить приложение.
3. Выполнить `php artisan migrate --force` (из одноразового job/init-контейнера или в pipeline до переключения трафика).
4. При необходимости — разовые команды: `storage:link`, кэш конфигурации/маршрутов/views: `config:cache`, `route:cache`, `view:cache` (только если это совместимо с вашим деплоем и динамическими путями).

---

## 7. Безопасность и сеть

1. Только необходимые порты наружу; админки и отладка закрыты.
2. HTTPS: терминация на reverse proxy (Traefik, nginx, cloud load balancer) с валидными сертификатами.
3. Обновления базовых образов по расписанию; сканирование на уязвимости (Trivy и аналоги) в CI.

---

## 8. Наблюдаемость и надёжность

1. Healthcheck для PHP/nginx (HTTP GET к маршруту вроде `/up` или отдельный lightweight endpoint).
2. Резервное копирование БД и критичных volumes по регламенту.
3. Лимиты CPU/RAM в compose/Kubernetes, политика `restart: unless-stopped` или аналог.

---

## 9. CI/CD (кратко)

1. Сборка образа в pipeline, пуш в registry с тегом (commit SHA + semver).
2. Прогон тестов до сборки образа.
3. Деплой: pull нового тега, `docker compose up -d` (или rolling update), миграции, проверка healthcheck, переключение трафика при blue-green, если используется.

---

## 10. Проверка после выката

1. Ответ приложения по HTTPS, редиректы HTTP → HTTPS.
2. Авторизация, критичные пользовательские сценарии, фоновые задачи (очередь), планировщик по логам.
3. Ошибки в логах контейнеров в первые часы после релиза.

---

## Варианты, которые часто лучше «всё в одном compose»

Кратко: когда что имеет смысл вместо или поверх описанного плана.

| Вариант | Зачем |
|--------|--------|
| **Управляемая БД и Redis** (RDS, Cloud SQL, ElastiCache и т.п.) | Резервные копии, обновления, отказоустойчивость и мониторинг «из коробки»; не держать Postgres в том же compose, что и приложение, если это не осознанный минимальный стенд. |
| **PaaS под Laravel** (Laravel Cloud, Forge + сервер, Railway, Render и аналоги) | Меньше ручной эксплуатации: деплой из git, сертификаты, масштабирование веба; плата за удобство вместо времени на Dockerfile и compose. |
| **Контейнеры в облаке без своих VM** (ECS/Fargate, Cloud Run, Azure Container Apps) | Платите за использование, автоскейл, изоляция; нет постоянной «сборки» ОС на VPS. |
| **Kubernetes** (EKS, GKE, k3s на нескольких нодах) | Имеет смысл при нескольких сервисах, zero-downtime rollouts, лимитах по командам и уже есть компетенция в k8s; для одного монолита часто избыточно. |
| **Гибрид с планом выше** | Собственный Docker-образ приложения + managed БД/Redis/очередь (SQS) + объектное хранилище для файлов — меньше stateful-сервисов в compose, проще бэкапы и восстановление. |
| **Octane / FrankenPHP** вместо классического php-fpm | Выше RPS при той же железке за счёт долгоживущих воркеров; другая модель деплоя и внимание к утечкам памяти в коде. |
| **Статика и фронт на CDN** | Отдавать `public/build` через CDN/edge — меньше нагрузки на PHP-контейнер. |

Итог: «docker compose на одной машине» — рабочая база для старта и внутренних сервисов; для продакшена с SLA чаще выигрывает сочетание **контейнер (или PaaS) для приложения** + **managed данные** + **нормальный CI/CD**, а не один файл compose со всем сразу.

---

## Связь с текущим репозиторием

Локально уже есть `docker-compose.postgres.yml` только для PostgreSQL. Для полного prod обычно добавляют отдельный compose-файл (или override) с приложением, веб-сервером и воркерами, либо выносят БД в managed-сервис и в compose остаётся только приложение и сопутствующие сервисы.
